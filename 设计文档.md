# 人脸签到系统 — 设计文档

## 一、项目概述

本系统是一个基于人脸识别的移动端签到系统。用户在手机端拍照上传后，服务器使用视觉神经网络完成人脸检测与识别，自动匹配用户身份并记录签到。系统部署在云服务器上，通过公网 IP 提供访问。

### 核心功能

| 功能 | 是否需要登录 | 说明 |
|------|:---:|------|
| 照片库管理（注册/查看/删除） | ✅ | 管理员录入用户基本信息及标准人脸照片 |
| 人脸签到 | ❌ | 拍照上传后自动检索匹配用户，无需设备绑定 |
| 签到记录查看 | ✅ | 按时间倒序分页展示签到历史 |

### 2.2 技术选型

| 层次 | 技术 | 选型理由 |
|------|------|----------|
| 后端框架 | **FastAPI** (Python) | 异步高性能，原生支持 Swagger 文档，适合 API 服务 |
| ASGI 服务器 | **Uvicorn** | 轻量级 ASGI 服务器，配合 FastAPI 提供异步 I/O |
| 数据库 | **SQLite** + SQLAlchemy ORM | 轻量免安装，适合中小规模部署；通过 WAL 模式提升并发读 |
| 认证 | **JWT** (HS256) | 无状态认证，减少服务端存储开销 |
| 人脸检测/识别 | **InsightFace** (buffalo_l) | 工业级精度，一站式提供检测+编码，基于 ONNX Runtime |
| 推理引擎 | **ONNX Runtime** (CPU) | 跨平台 CPU 推理，无需 GPU |
| 移动端 | **Flutter** | 跨平台，一套代码同时支持 Android/iOS |
| 反向代理 | **Nginx** | 静态文件服务、请求限流、连接复用 |
| 部署 | **华为云 ECS** | 公网 IP 访问，弹性扩展 |

## 四、模型选择与设计

### 4.1 人脸识别 Pipeline

```
输入图片 → [人脸检测] → [人脸对齐/裁剪] → [特征提取(512维)] → [余弦相似度检索]
```

### 4.2 模型选型

| 环节 | 模型/方案 | 说明 |
|------|-----------|------|
| **人脸检测** | InsightFace `buffalo_l` 内置检测器 | 基于 RetinaFace 的高精度人脸检测，输出 bbox + 5 关键点 |
| **人脸编码** | InsightFace `buffalo_l` 内置 ArcFace | 输出 512 维归一化特征向量，类内紧凑、类间分散 |
| **特征检索** | 全量余弦相似度比对 | 将签到照片特征与库中所有特征逐一计算余弦相似度，取最高分且超过阈值 |

### 4.3 选型理由

| 对比维度 | InsightFace (ArcFace) | OpenCV DNN (FaceNet) | dlib (ResNet) |
|----------|-----------------------|----------------------|---------------|
| 检测精度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 识别精度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| CPU 推理速度 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| 安装便捷性 | ⭐⭐⭐⭐ (pip install) | ⭐⭐⭐⭐⭐ | ⭐⭐ (需编译) |
| 社区活跃度 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |

**选择 InsightFace** 的核心原因：
- 一站式提供检测 + 编码，无需额外整合多个模型
- ArcFace 在 LFW、MegaFace 等基准上表现优异
- 纯 ONNX Runtime 推理，CPU 即可满足需求，无需 GPU
- `pip install insightface` 即可安装，零编译成本

### 4.4 核心参数配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 模型名称 | `buffalo_l` | InsightFace 预训练模型包 |
| 推理设备 | CPU (ONNX Runtime) | 云服务器无 GPU 场景 |
| 特征维度 | 512 | float32 归一化向量 |
| 相似度阈值 | 0.6 | 余弦相似度 ≥ 0.6 判定为匹配成功 |
| 检测阈值 | 0.5 | 人脸检测置信度最低要求 |
| 最小人脸尺寸 | 30px | 忽略过小的人脸区域 |
| 图像处理 | OpenCV (BGR→RGB) | 统一颜色空间转换 |

## 五、API 接口设计

### 5.1 接口总览

| 方法 | 路径 | 认证 | 说明 |
|------|------|:---:|------|
| POST | `/api/register` | ❌ | 用户注册（含标准照上传 + 人脸编码） |
| POST | `/api/login` | ❌ | 登录获取 JWT Token |
| POST | `/api/logout` | ✅ | 登出（无状态） |
| POST | `/api/checkin` | ❌ | 拍照上传签到（人脸检索） |
| GET | `/api/checkin/records` | ✅ | 分页查询签到记录 |
| GET | `/api/users` | ✅ | 用户列表（照片库管理） |
| DELETE | `/api/users/{id}` | ✅ | 删除用户 |

### 5.2 关键接口流程

#### 注册流程

```
手机端 ──POST /api/register──▶ 服务器
         (multipart: 基本信息 + 标准照)
                                    │
                                    ▼
                              保存照片到磁盘
                                    │
                                    ▼
                              人脸检测 (FaceDetector)
                               ↓ 检测到人脸
                              人脸编码 (FaceEncoder → 512维向量)
                               ↓ 编码成功
                              创建 User 记录
                              创建 FaceFeature 记录
                              返回成功
                              （任一步骤失败则回滚）
```

#### 签到流程

```
手机端 ──POST /api/checkin──▶ 服务器
         (multipart: 现场照片)
                                    │
                                    ▼
                              保存照片到磁盘
                                    │
                                    ▼
                              人脸检测 → 人脸编码 (512维向量)
                                    │
                                    ▼
                              遍历 face_features 表
                              逐一计算余弦相似度
                                    │
                          ┌─────────┴─────────┐
                          ▼                   ▼
                    最高分 ≥ 0.6          最高分 < 0.6
                    创建成功记录          创建失败记录
                    返回匹配用户信息      返回识别失败
```

## 六、高并发处理方案

### 6.1 面临的挑战

- 签到高峰期（如会议开始前）可能出现瞬时大量请求
- 人脸推理（检测 + 编码）是 **CPU 密集型** 操作，单次约 200-500ms
- 默认的单进程同步模型无法充分利用多核 CPU

### 6.2 应对策略

| 策略 | 方案 | 说明 |
|------|------|------|
| **ASGI 异步框架** | FastAPI + Uvicorn | 异步 I/O 处理文件上传，不阻塞主线程 |
| **多 Worker 部署** | Uvicorn `--workers 4` | 利用多核 CPU，每个 worker 独立加载模型 |
| **请求限流** | Nginx `limit_req` | 限制单 IP 请求频率，防止恶意刷接口 |
| **数据库优化** | SQLite WAL 模式 | Write-Ahead Logging，读不阻塞写，提升并发 |
| **连接池复用** | Nginx `keepalive` | 减少 TCP 握手开销 |
| **人脸模型单例** | 模块级懒加载 | 每个 worker 启动时加载一次模型，请求间复用 |

### 6.3 部署配置示例

```nginx
# Nginx 限流配置
limit_req_zone $binary_remote_addr zone=checkin:10m rate=5r/s;

location /api/checkin {
    limit_req zone=checkin burst=10 nodelay;
    proxy_pass http://backend;
}
```

```bash
# 多 Worker 启动
uvicorn main:app --host 0.0.0.0 --port 8000 --workers 4
```

## 七、安全设计

| 方面 | 方案 |
|------|------|
| 密码存储 | bcrypt 哈希，不可逆 |
| 身份认证 | JWT HS256，24h 过期 |

| 敏感信息 | `SECRET_KEY` 通过环境变量注入，不硬编码 |

## 十、后续优化方向

1. **特征检索优化**：当用户量增大时，引入 **FAISS** 向量索引替代全量遍历比对，将检索复杂度从 O(N) 降至 O(log N)
2. **数据库升级**：迁移至 PostgreSQL，支持更高并发写入
3. **模型加速**：使用 ONNX Runtime 的 INT8 量化，减少推理延迟
4. **活体检测**：增加眨眼/张嘴动作检测，防止照片欺骗
5. **异步推理队列**：引入 Celery + Redis，将人脸推理从请求线程中解耦
